# http_survey — бенчмарк двух подходов к ограничению конкурентности

Учебная задача: есть список URL-ов, нужно опросить каждый и вернуть HTTP-статусы
**в том же порядке**, в котором URL-ы были переданы. Кол-во одновременно идущих
запросов ограничено параметром `parallel`.

В репозитории реализованы и сравниваются **две** реализации функции
`Survey(ctx, urls, parallel, isMock)`:

1. **Семафор** (`http_survey/internal/semaphore`) — на каждый URL стартует отдельная
   горутина, конкурентность ограничивается буферизированным каналом-семафором
   `chan struct{}` ёмкости `parallel`.
2. **Пул воркеров** (`http_survey/internal/workerpool`) — заранее создаётся `parallel`
   долгоживущих горутин-воркеров, читающих задачи из общего входного канала.

Цель — сравнить обе реализации по времени выполнения, потреблению памяти и
количеству аллокаций при разных размерах входа и уровнях параллелизма.

## Описание задачи

На вход подаётся список URL-ов и лимит параллелизма `parallel`. Функция
`Survey(ctx, urls, parallel, isMock)` должна обойти все URL-ы, выполнить HTTP
GET для каждого адреса и вернуть срез HTTP-статусов той же длины, что и входной
список.

Основные требования:

- сохранить порядок результатов: `results[i]` относится к `urls[i]`;
- ограничить количество одновременно выполняемых HTTP-запросов;
- корректно работать с дублирующимися URL-ами;
- поддерживать отмену через `context.Context`;
- не допускать гонок при записи результатов и работе с общими структурами;
- сравнить две модели конкурентности: горутина на задачу с семафором и
  фиксированный пул воркеров.

Дополнительно в проект добавлены кеширование, retry budget и `singleflight`.
Они приближают учебную задачу к реальному HTTP-клиенту: одинаковые URL-ы не
должны лишний раз ходить в сеть, а ретраи не должны усиливать нагрузку на
внешний сервис во время деградации.

## Использованные паттерны

**Semaphore / bounded concurrency.** В реализации `semaphore` на каждую задачу
создаётся отдельная goroutine, а буферизированный канал `chan struct{}` задаёт
верхнюю границу параллелизма. Это простой способ ограничить число одновременно
выполняемых запросов без отдельной очереди задач.

**Worker Pool.** В реализации `workerpool` заранее запускается `parallel`
долгоживущих worker-горутин. Задачи передаются через канал `chan job`, а каждый
worker пишет результат в заранее известную позицию `results[idx]`. Такой подход
уменьшает количество создаваемых goroutine и аллокаций на больших пачках URL-ов.

**Fan-out / fan-in.** Оба варианта сначала распределяют работу между несколькими
goroutine, а затем собирают результат обратно в один срез. `sync.WaitGroup`
используется как точка синхронизации, чтобы дождаться завершения всех задач
перед возвратом результата.

**Thread-safe sharded cache.** Пакет `cache` хранит `url -> statusCode` в
нескольких шардах, у каждого шарда свой `sync.RWMutex` и `map`. Шард для URL
выбирается через `crc32.ChecksumIEEE([]byte(url)) % nodesLen`, поэтому
конкурентные обращения к разным шардам меньше блокируют друг друга.

**Singleflight.** В `internal/client` используется `golang.org/x/sync/singleflight`.
Если несколько goroutine одновременно запрашивают один и тот же URL, фактический
HTTP-запрос выполняется один раз, а остальные goroutine получают общий результат.
Это защищает от cache stampede при холодном кеше.

**Retry Budget.** Пакет `internal/retrybudget` ограничивает количество повторных попыток.
Бюджет пополняется после успешных ответов и расходуется перед retry. Такой
подход лучше простого "retry N раз", потому что не даёт ретраям бесконтрольно
усиливать нагрузку во время массовых ошибок.

**Interface-based client.** Обе реализации зависят от интерфейса `Getter`, а не
от конкретного HTTP-клиента. Поэтому в тестах и бенчмарках можно использовать
`httpClientMock`, не меняя код конкурентности.

**Benchmark-driven comparison.** Сравнение реализаций вынесено в общий
benchmark-пакет. Один и тот же набор входных данных прогоняется через обе
реализации, после чего сравниваются `ns/op`, `B/op` и `allocs/op`.

## Как запустить тесты

Все команды выполняются из корня репозитория `go-katas`.

### 1. Корректностные тесты под детектором гонок

```powershell
go test ./http_survey/bench/ -run=TestSurvey -race -v
```

Что проверяется:

- **`TestSurveyCorrectness`** — табличный тест: пустой вход, один URL,
  дубликаты (включаются кеш-хиты), `parallel > len(urls)`, большие списки.
  Контролируется длина результата и значения статусов для **каждой** реализации.
- **`TestSurveyEquivalence`** — `slices.Equal(gotSemaphore, gotWorkerPool)`:
  обе реализации обязаны вернуть один и тот же срез статусов (включая порядок).

Флаг `-race` включает детектор гонок: ловит конкурентные записи в общий
`results` и т. п.

### 2. Бенчмарки

Короткий прогон (для отладки):

```powershell
go test ./http_survey/bench/ -run=^$ -bench=. -benchmem -count=3 -benchtime=200ms
```

Полный прогон с записью в файл (для последующего анализа `benchstat`):

```powershell
go test ./http_survey/bench/ -run=^$ -bench=. -benchmem -count=10 -benchtime=300ms > new.txt
```

### 3. Сводка через `benchstat`

```powershell
go install golang.org/x/perf/cmd/benchstat@latest
& "$(go env GOPATH)\bin\benchstat.exe" new.txt
```

Сравнение двух прогонов (например, до и после оптимизации):

```powershell
& "$(go env GOPATH)\bin\benchstat.exe" old.txt new.txt
```

### 4. Профилирование

```powershell
go test ./http_survey/bench/ -run=^$ `
  -bench='semaphore/urls=1000/uniq=1000/par=64' `
  -benchtime=5s -cpuprofile cpu.out -memprofile mem.out
go tool pprof -http=:8080 cpu.out
```

## Методика бенчмарка

Бенчмарк (`BenchmarkSurvey`) гоняет каждую реализацию по матрице конфигов:

- **`urls`** — длина входного списка (100 и 1000);
- **`unique`** — сколько различных URL-ов в списке (управляет долей кеш-хитов);
- **`parallel`** — лимит одновременных запросов (1, 8, 32, 64, 128, 256).

В качестве клиента используется `httpClientMock`, который **мгновенно** возвращает
`200`. Это умышленно: при реальной сети мы бы мерили latency сети, а не разницу
между примитивами синхронизации. Сейчас бенчмарк показывает именно стоимость
управления конкурентностью.

Для исключения влияния порядка обхода `map` (в Go он рандомен) реализации
перебираются по слайсу `[]struct{name, fn}` — порядок прогона стабилен.

### Почему семафор аллоцирует больше

На первый взгляд код семафорной версии простой и «ничего лишнего» в нём нет.
Но цифры показывают разницу почти **2×** по байтам и **~47×** по числу
аллокаций. Разберём на кейсе `urls=1000, uniq=1000, par=8`:

| | semaphore | workerpool | разница |
|---|---:|---:|---:|
| `B/op` | 232 KiB | 115 KiB | **+117 KiB** |
| `allocs/op` | 2028 | 43 | **+1985** |

То есть на 1000 URL семафор делает **примерно 2 «лишних» аллокации на каждый
URL** общим объёмом ~120 байт. Источник — два замыкания в горячем цикле:

```go
for i, url := range urls {
    semaphore <- struct{}{}

    wg.Go(func() {                          // ← замыкание №1 (внешнее)
        defer func() { <-semaphore }()      // ← замыкание №2 (внутреннее)

        statusCode, ok := cache.Get(url)
        if !ok {
            statusCode = client.Get(ctx, url)
            cache.Set(url, statusCode)
        }
        results[i] = statusCode
    })
}
```

1. **Внешнее замыкание** для `wg.Go(...)` — это объект в куче, в котором лежат
   все захваченные переменные: `i`, `url`, `cache`, `client`, `ctx`,
   `semaphore`, `results`. Он утекает на heap, потому что передаётся в
   горутину и переживает функцию. Получается одна аллокация ~80–100 байт на
   каждый URL.
2. **Внутреннее замыкание** в `defer func() { <-semaphore }()` — отдельный
   объект на куче (~16–24 байта) на каждый URL.

Арифметика сходится: `(232 − 115) KiB / 1000 URL ≈ 117 байт на URL ≈ 60 байт
на одно замыкание × 2`. Цифра аллокаций (`2028 ≈ 2 × 1000 + 28`) тоже бьётся
с этой моделью.

У воркер-пула таких затрат **нет**: замыкание создаётся ровно `parallel` раз
(8 на 1000 URL — то есть в 125× меньше), а в горячем цикле выполняется
только `chanInput <- job{idx, url}` — копирование 24-байтного значения в уже
выделенный кольцевой буфер канала, без аллокаций на куче. Поэтому
`B/op` и `allocs/op` у воркер-пула растут как `O(parallel)`, а не
`O(len(urls))`.

Убедиться можно самостоятельно через escape-анализ компилятора:

```powershell
go build -gcflags='-m -m' ./http_survey/internal/semaphore/ 2>&1 | rg "func literal|escapes to heap"
```

в выводе будут строки вроде `func literal escapes to heap` — это и есть наши
два замыкания. Или через профиль аллокаций:

```powershell
go test ./http_survey/bench/ -run=^$ -bench='semaphore/urls=1000/uniq=1000/par=8' `
    -benchmem -memprofile mem.out -benchtime=2s
go tool pprof -alloc_space -top mem.out
```

На вершине списка будут `runtime.newobject` из `semaphore.Survey.func1` —
функции замыканий.

Это, к слову, не баг реализации, а **свойство паттерна**: одна горутина
на единицу работы → O(N) аллокаций по определению.

## Выводы

1. **Аллокации — ключевое преимущество воркер-пула.** Семафорная версия делает
   `~2 × len(urls)` аллокаций (по горутине и замыканию на каждый URL), пул —
   `O(parallel)` независимо от длины списка. На 1000 URL это **~47×** меньше
   аллокаций (43 против 2028 при `par=8`). Для долго живущих программ и
   GC-нагруженных сценариев это самый значимый аргумент.

2. **Время — пул быстрее в большинстве случаев**, но с инверсией при
   `parallel ≥ len(urls)`. Если воркеров заведомо больше, чем задач, пул
   платит за лишние горутины-«пустышки», и семафор обходит его (например,
   `urls=100 par=128`: 146 µs vs 283 µs). На реальной задаче это значит, что
   `parallel` стоит подбирать **по числу URL и латенси сети**, а не «с запасом».

3. **Кеш сглаживает разницу.** В режиме `uniq=50, urls=1000` 95% «запросов» —
   это попадания в кеш, обе реализации упираются в `RWMutex` кеша, и время
   почти выравнивается (553 µs vs 624 µs). Это место — кандидат на
   `singleflight` (пункт 7 ДЗ): при холодном старте на одинаковый URL воркеры
   могут параллельно сходить в `client.Get`, чего хочется избежать.

4. **Реализации эквивалентны по контракту.** `TestSurveyEquivalence` со
   строгим `slices.Equal` подтверждает: обе версии возвращают один и тот же
   срез статусов в одном и том же порядке. Гонок под `-race` нет.

### Эмпирическое правило выбора

- `parallel ≪ len(urls)` (типичный случай — лимит исходящих соединений) →
  **workerpool** выигрывает по обеим осям.
- `parallel ≥ len(urls)` (короткие пачки, всё запускается «сразу») →
  **semaphore** проще и быстрее.
- Большая доля кеш-хитов → реализация почти не важна, узкое место — кеш.
